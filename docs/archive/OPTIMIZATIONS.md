# Оптимизации для слабых ноутбуков

## Обзор

Field Mapper теперь может работать на ноутбуках с ограниченными ресурсами (4GB RAM, 2 ядра CPU).

## Что оптимизировано

### 1. raster_service.py — Windowed Reading

**Было:** Читает весь GeoTIFF в память, потом обрезает по полю и уменьшает.

**Стало:** 
- Сначала определяет bounding box поля
- Читает только нужную область с нужным разрешением
- Дополнительное уменьшение если данные всё ещё большие

**Экономия RAM:** 40-60% для больших GeoTIFF файлов

### 2. SQLite WAL Mode

**Было:** Обычный SQLite (блокировка при параллельных запросах)

**Стало:**
- WAL (Write-Ahead Logging) режим
- 64MB кэш
- Параллельное чтение + запись

**Результат:** Нет блокировок при нескольких пользователях

### 3. orthomosaic_service.py — Векторизация

**Было:** Попиксельный цикл Python для blending (10000×10000 = 100 млн итераций)

**Стало:** Numpy векторизованные операции (всё на C уровне)

**Ускорение:** 5-10x быстрее при склейке ортомозаики

### 4. Docker Resource Limits

```yaml
app:     2G RAM, 1.5 CPU
worker:  3G RAM, 2.0 CPU (1 worker)
redis:   256M RAM, 0.5 CPU
nginx:   128M RAM, 0.25 CPU
```

**Результат:** Нет OOM kills, предсказуемое потребление ресурсов

## Минимальные требования

| Ресурс | Без ортомозаики | С ортомозаикой |
|--------|-----------------|----------------|
| RAM | 4 ГБ | 8 ГБ |
| CPU | 2 ядра | 4 ядра |
| Диск | 10 ГБ | 20 ГБ |

## Рекомендации

### Для слабых ноутбуков (4GB RAM)

Запускать только основные сервисы:
```bash
docker compose up app redis nginx
# без worker — отключаем обработку ортомозаики
```

### Для комфортной работы (8-16GB RAM)

Полный стек:
```bash
docker compose up -d
```

### Мониторинг ресурсов

```bash
docker stats
```

Показывает реальное потребление RAM/CPU каждым контейнером.
